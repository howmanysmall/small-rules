`small-rules/prefer-expect-assertions` reports tests that omit an assertion guard. Its source metadata describes it as: `Enforce expect assertion guards in Jest tests and prefer expect.assertions(n) over expect.hasAssertions() when the count is known`.

## Rule Details

This rule is configured as `small-rules/prefer-expect-assertions` and reports the concrete syntax covered by its implementation in `src/rules/prefer-expect-assertions.ts`. It requires each test to begin with `expect.assertions(<number>)` or `expect.hasAssertions()`, and prefers `expect.assertions(n)` when the number of assertions is statically known.

<Aside type="note">

Rule message:

```text
Every test should have either `expect.assertions(<number of assertions>)` or `expect.hasAssertions()` as its first expression
```

</Aside>

## Configuration

```json
{
	"plugins": ["@pobammer-ts/small-rules"],
	"rules": {
		"small-rules/prefer-expect-assertions": "error"
	}
}
```

This rule accepts options including `additionalAssertionFunctions`, `additionalExpectCallNames`, `onlyFunctionsWithAsyncKeyword`, and `onlyFunctionsWithExpectInCallback`.

## Examples

<RuleExample type="fail" title="Fails">

```ts
it("adds numbers", () => {
	expect(add(1, 2)).toBe(3);
});
```

</RuleExample>

<RuleExample type="pass" title="Passes">

```ts
it("adds numbers", () => {
	expect.assertions(1);
	expect(add(1, 2)).toBe(3);
});
```

</RuleExample>
